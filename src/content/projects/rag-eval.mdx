# RAG Evaluation Pipeline

Building a production RAG system is relatively straightforward. Building a RAG system you can confidently iterate on — because you can measure what changes — is significantly harder. This project is an automated evaluation framework that makes that measurement systematic.

## Problem Statement

Standard LLM benchmarks do not capture the failure modes specific to RAG systems. A model can be fluent, factually accurate on general knowledge, and still fail completely when asked to answer questions grounded in a private document corpus.

RAG introduces three independent failure modes:

1. The retriever fails to surface the relevant documents (retrieval failure)
2. The generator hallucinates facts not present in the retrieved context (faithfulness failure)
3. The generator produces an answer inconsistent with the ground truth (correctness failure)

Conflating these three failures makes debugging impossible. This pipeline separates them.

## Architecture

The pipeline consists of three stages executed in sequence per query:

**Stage 1 — Retrieval Evaluation**

Given a query and a set of ground-truth relevant document IDs, compute recall@k for k ∈ {1, 3, 5, 10}. This stage runs entirely offline against a labeled dataset and does not require LLM inference. Results are stored as a parquet file partitioned by retriever configuration.

**Stage 2 — Faithfulness Scoring**

For each (query, retrieved context, generated answer) triple, we prompt Claude 3.5 Sonnet with a faithfulness rubric: "Does this answer contain only claims that can be directly supported by the provided context?" The model returns a binary score plus a list of unsupported claims. This catches hallucinations at the sentence level.

**Stage 3 — Answer Correctness**

Correctness is measured via pairwise comparison: we compare the generated answer against a reference answer written by a human annotator. Rather than asking for a 1–5 score (which suffers from anchor effects), we ask "which of these answers is more correct and complete?" and compute an Elo-style rating over the full dataset.

## Dataset Construction

The evaluation dataset was built from 500 questions drawn from internal Confluence pages. Questions were written by annotators who were shown a document and asked to write questions whose answers could only be found in that specific document. This process surfaces retrieval-specific failure modes that public benchmarks miss.

We maintain three splits:
- `train` (300 examples): used for prompt engineering and threshold calibration
- `val` (100 examples): used for model selection during development  
- `test` (100 examples): held out until final reporting

## Results

On our internal corpus:

| Metric | Baseline (BM25) | Improved (Hybrid + Rerank) |
|--------|-----------------|---------------------------|
| Recall@5 | 0.71 | 0.89 |
| Faithfulness | 0.83 | 0.91 |
| Correctness (Elo) | 1200 | 1347 |

The hybrid retriever with cross-encoder reranking improved recall@5 by 25%, which propagated directly into faithfulness and correctness scores since the generator had better context to work with.

## Lessons Learned

Faithfulness scoring with an LLM judge is surprisingly reliable when the rubric is binary (supported/not supported) rather than graded. Binary rubrics reduce inter-rater variance and make the prompt easier to calibrate.

Retrieval recall is the highest-leverage place to invest engineering effort. The generator quality matters, but a 10% improvement in recall@5 consistently yields a larger downstream quality gain than switching from one frontier model to another.

Human annotation quality is the bottleneck. We iterated through three annotation rubrics before arriving at one with sufficient inter-annotator agreement (Cohen's κ > 0.75) to be useful as ground truth.
