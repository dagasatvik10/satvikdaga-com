# Getting Started with LLM Evaluation

Evaluating large language models is one of the most important — and most misunderstood — parts of building production AI systems. In this post I want to walk through the core concepts, common pitfalls, and practical patterns I've learned building evaluation pipelines at scale.

## Why Evaluation Is Hard

The fundamental challenge with LLM evaluation is that language is ambiguous. When a model generates text, there is rarely a single correct answer. A factual question like "What is the capital of France?" has an obvious ground truth, but most real-world tasks — summarization, code generation, reasoning — do not.

This ambiguity means we need multiple evaluation strategies layered together, each capturing a different signal.

## The Three Pillars of LLM Evaluation

### 1. Reference-Based Metrics

These compare model output against a gold-standard reference. Classic examples include BLEU for translation and ROUGE for summarization. They are fast and deterministic, but they penalize valid paraphrases and reward surface-level overlap over semantic correctness.

Use reference-based metrics when you have high-quality labeled data and the output space is constrained — for example, extractive question answering where the answer must appear verbatim in a source document.

### 2. Model-Based Evaluation

LLM-as-judge has become the dominant approach for open-ended generation tasks. You prompt a strong model (GPT-4, Claude 3.5 Sonnet) with the task, the model output, and a rubric, then ask it to score or compare responses.

The key insight here is that you are not measuring absolute quality — you are measuring relative quality. Pairwise comparisons ("which of these two responses is better?") are more reliable than absolute scoring because they reduce position bias and anchor effects.

### 3. Behavioral Testing

Rather than scoring outputs, behavioral tests assert that specific invariants hold. A model that correctly answers a question should also correctly answer semantically equivalent rephrases. A model that refuses a harmful request should refuse similar harmful requests.

Behavioral testing is especially valuable for safety and alignment work, where you want to verify that fine-tuning did not break existing capabilities.

## Building a RAG Evaluation Pipeline

Retrieval-augmented generation introduces an additional dimension: not only must the generated answer be correct, but the retrieved context must be relevant. This requires evaluating three things independently:

1. **Retrieval recall**: Did the retriever surface the relevant documents?
2. **Context faithfulness**: Does the generated answer only rely on facts present in the retrieved context?
3. **Answer correctness**: Is the answer factually accurate given the question?

I use RAGAS for automated scoring of faithfulness and answer relevancy, combined with human spot-checks on a stratified sample of queries.

## Practical Advice

Start simple. A spreadsheet with 50 hand-labeled examples and a manual evaluation rubric will teach you more about your failure modes than any automated pipeline. Build automation once you understand what you are measuring.

Track regressions, not just absolute scores. A new model checkpoint that scores 0.85 on your eval suite is only useful if you know it previously scored 0.82. Version your eval datasets and store results in a structured format alongside your model checkpoints.

Evaluate on the distribution you care about. The biggest mistake I see is evaluating on a benchmark that looks like the training data rather than the actual deployment queries. Collect real user queries early and build your eval set from them.
